# Mention API

`@mention/backend` is an Express 5 service that powers the Mention frontend, the mobile apps, the MCP server, and the federation surface. It speaks REST over HTTPS for request/response work and Socket.io for real-time updates. Authentication is delegated to Oxy: every authenticated request carries an Oxy access token in `Authorization: Bearer <token>`, verified by `@oxyhq/core`'s `auth()` middleware.

MCP-specific routes (`/mcp/oauth/*`, `/mcp/bundles/*`, `/mcp/connections`) also accept a Mention MCP access JWT with the same Bearer header; see [`packages/mcp/README.md`](../packages/mcp/README.md).

## Base URL

```
https://api.mention.earth
```

Local development defaults to `http://localhost:3000`. The federation endpoints (`/.well-known/*`, `/ap/*`) must additionally be reachable on the `FEDERATION_DOMAIN` — see [Fediverse](/developers/docs/mention/fediverse).

## Authentication

| Type | Header | Use case |
|------|--------|----------|
| Oxy user token | `Authorization: Bearer <oxy_access_token>` | First-party Mention clients (web, mobile). |
| HTTP signature | `Signature: keyId=..., signature=...` | Inbound fediverse activities (verified against `FederatedActor` public keys). |

There is no Mention-specific signup or login flow. Users authenticate against Oxy (`@oxyhq/services` in-app or the standalone Auth app on the web) and the resulting access token is sent to the Mention API. The backend pulls user metadata from Oxy as needed.

Public endpoints either skip auth entirely (e.g. trending, hashtag pages) or use the `optionalAuth` middleware that attaches the user when present but does not require it. Write endpoints always require auth.

## Conventions

- **Errors** — every error response follows `{ error: 'CODE', message: 'human readable' }` with the appropriate HTTP status.
- **Pagination** — list endpoints accept `cursor` (opaque) and `limit` query params; responses include `nextCursor` when more data exists. Feed endpoints expose dual cursors (`score` for ranked feeds, `chrono` for time-ordered feeds) handled by `CursorBuilder`.
- **Rate limiting** — Redis-backed via `express-rate-limit` + `rate-limit-redis`. Brute-force protection sits on top via `express-slow-down`. Exceeded limits return `429` with `Retry-After`.
- **Timestamps** — ISO 8601 (`2025-01-15T12:34:56.789Z`).
- **IDs** — Mongo ObjectIDs serialized as strings.
- **DTOs** — request and response shapes match the interfaces exported from `@mention/shared-types`.

## Feed endpoints (MTN)

`/feed/mtn` is the unified read endpoint backed by the MTN feed engine. The feed identity is encoded in the request — `descriptor`, `cursor`, `limit` — and the dispatcher looks up the right `FeedAPI` implementation from `FeedAPIRegistry`.

| Method | Path | Description |
|--------|------|-------------|
| GET | `/feed/mtn` | Fetch a feed page. Query: `descriptor`, `cursor`, `limit`. Returns `SlicedFeedResponse`. |
| GET | `/feed/mtn/peek` | Peek the latest item without consuming a cursor — drives the "new posts" pill. |
| POST | `/feed/mtn/interactions` | Record impressions / interactions for ranking signal. |
| GET | `/feed/replies/:parentId` | Replies feed for a post. |
| GET | `/feed/user/:userId` | Author feed (with public-profile caching). |
| GET | `/feed/user/:userId/pinned` | Pinned post for a user. |
| GET | `/feed/item/:id` | Single hydrated feed item by ID. |
| POST | `/feed/reply` | Create a reply. |
| POST | `/feed/repost` | Create a repost. |
| DELETE | `/feed/:postId/repost` | Remove a repost. |
| POST | `/feed/like` | Like a post. |
| POST | `/feed/unlike` | Unlike a post. |
| POST | `/feed/:postId/save` | Save a post to bookmarks. |
| DELETE | `/feed/:postId/save` | Unsave. |

Feed responses use `HydratedPost` from `@mention/shared-types`, which bundles the post content with its actor summary, attachment bundle, viewer state (`isLiked`, `isReposted`, ...), permissions, engagement counts, and feed context (`reason`, `position`).

When a notification is persisted, the server attempts an FCM push to every registered token for the recipient. Push notifications require `FIREBASE_PROJECT_ID` and `FIREBASE_SERVICE_ACCOUNT_BASE64` in the backend environment.

## Real-time (Socket.io)

The Socket.io server uses `@socket.io/redis-adapter` so it scales horizontally behind a Redis pub/sub channel. Clients connect with their Oxy bearer token and subscribe to rooms keyed by `oxyUserId`.

Common events emitted to clients:

- `notification:new` — payload is the freshly-persisted `Notification`.
- `post:stats` — engagement count updates for posts currently in view.
- `feed:invalidate` — instructs the client to refetch a specific feed descriptor.
- `room:participant_joined`, `room:participant_left` — for live LiveKit rooms.

Clients send `feed:subscribe` / `feed:unsubscribe` to control which feeds the server pushes invalidations for, keeping fan-out cost bounded.

## Errors

Common error codes the API can return:

| HTTP | Code | Meaning |
|------|------|---------|
| 400 | `VALIDATION_FAILED` | Request body failed `express-validator` checks. Details under `errors`. |
| 401 | `UNAUTHENTICATED` | Missing or invalid Oxy bearer token. |
| 403 | `FORBIDDEN` | Authenticated, but not allowed (e.g. blocked, muted, or non-admin hitting `/admin`). |
| 404 | `NOT_FOUND` | Resource not found. |
| 409 | `CONFLICT` | State conflict (e.g. already liked). |
| 422 | `INVALID_STATE` | Request was well-formed but the action is not legal in the current state. |
| 429 | `RATE_LIMITED` | Rate or brute-force limit exceeded. `Retry-After` set. |
| 500 | `INTERNAL_ERROR` | Unhandled server error. Logged with Pino + request ID. |
